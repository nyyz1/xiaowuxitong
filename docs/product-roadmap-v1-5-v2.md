# 校务系统 V1.5 / V2 功能路线图

## 定位

这份路线图用于指导 V1 试点后的下一阶段建设。2026-05-11 起，申请审批已经作为 V1.5 Step 8 进入代码和数据库；本文剩余建议主要用于继续讨论数据质量、导入预检、常用导出、学生一页通等后续增强。任何新的 V1.5 或 V2 功能继续进入开发前，仍应先更新 `memory-bank/prd.md`、`memory-bank/implementation-plan.md` 和对应验收标准。

当前系统已经形成七个核心业务模块：

| 模块 | 当前价值 |
| --- | --- |
| 用户权限 | 支持系统管理员、校领导、部门领导、年级管理员、政教工作人员、教务工作人员、行政办公人员、后勤办公人员、一线教师等角色边界 |
| 学校结构 | 支持年级、班级、部门、学科和届别滚动 |
| 师生档案 | 支持师生信息维护、动态字段、导入、导出、误录修正 |
| 往届存档 | 支持归档学生查询、维护、导入、导出 |
| 常规检查 | 支持学生量化和教师量化的项目、录入、查询、修正 |
| 申请审批 | 支持教师提交报修、打印和其他申请，并按审批职责通过或驳回 |
| 统计导出 | 支持检查统计、Excel/CSV 导出和审计 |

因此下一阶段的重点不是“马上做成大而全平台”，而是把现有闭环从“能录、能查、能导出”提升为“更省事、更准确、更可追溯、更方便领导决策”。

## 成熟校务系统对标结论

对标 PowerSchool SIS、Infinite Campus、Blackbaud Student Information System、openSIS 和 Ed-Fi 数据标准，成熟系统高频能力通常包括：

| 能力域 | 成熟系统常见做法 | 对本项目的启发 |
| --- | --- | --- |
| 学生信息主档 | 学生基础信息、家庭联系、状态、历史记录集中管理 | 当前已具备基础档案，应补齐学生一页通和变更历史 |
| 考勤与请假 | 日常出勤、迟到早退、缺勤、请假和统计 | 可作为 V1.5 轻量扩展，但不要和常规检查混成同一概念 |
| 成绩、课程与课表 | 课程安排、任课、成绩册、成绩单 | 价值高但范围大，适合 V2 |
| 行为与德育 | 纪律、奖惩、行为事件、成长记录 | 可由现有常规检查数据逐步演化 |
| 门户与沟通 | 家长/学生门户、公告、消息、移动端访问 | 先做校内通知和待办，家长/学生端暂缓到 V2 |
| 报表与分析 | 自定义报表、保存筛选、导出模板、统计看板 | 与当前架构最贴合，应作为 V1.5 重点 |
| 数据治理 | 权限、审计、导入校验、数据标准、接口 | 当前已有基础，应强化数据质量和安全运维 |

## V1.5 路线

V1.5 只选择贴合当前架构、低风险、高收益的改进，目标是在不重构系统、不引入复杂新基础设施的前提下提升真实试点可用性。

| 优先级 | 方向 | 建议能力 | 为什么先做 | 进入开发前需要确认 |
| --- | --- | --- | --- | --- |
| P0 | 数据质量 | 缺失字段、重复身份证号、异常班级/年级、孤立引用、导入预检、错误行修复 | 直接降低真实数据导入和长期维护风险 | 哪些字段是必填、哪些错误允许暂存 |
| P0 | 导入体验 | 上传后先预检，不立即写入；展示新增、更新、失败、警告；支持下载错误行 | 真实学校数据来自 Excel，预检能避免批量污染数据 | 是否允许部分成功，还是必须整表通过 |
| P1 | 常用筛选与导出模板 | 保存常用筛选条件、字段显示开关、常用导出模板 | 减少重复操作，帮助领导和年级办公室快速拿数据 | 模板是个人级、角色级还是全校共享 |
| P1 | 学生一页通 | 一个学生页面集中展示基础档案、班级/宿舍、状态、检查记录、导出入口 | 成熟 SIS 的高频入口，适合领导、年级主任、班主任快速了解学生 | 首版是否包含监护人、检查历史、请假摘要 |
| P2 | 轻量考勤/请假 | 班级日常出勤登记、请假状态、缺勤统计、导出 | 和学校日常管理高度相关，但可先做轻量版 | 是否按天、半天、课节，是否需要审批流 |
| P2 | 通知与待办 | 导入失败提醒、待复核数据、备份失败提醒、系统公告 | 先服务校内人员，不触碰家长端账号体系 | 是否需要消息已读、是否需要按角色推送 |
| P2 | 安全与运维强化 | 敏感字段脱敏、导出原因、登录失败限制、备份健康检查、关闭或收紧兜底账号 | 真实数据越多，越需要可控边界 | 生产环境是否保留 Bootstrap fallback |

### V1.5 建议验收

- 教务工作人员或具备导入权限的账号能在导入前看到新增、更新、失败和警告数量。
- 导入失败行能被定位到具体行号、字段和原因。
- 校领导或年级管理员能保存至少一组常用筛选并复用导出。
- 操作人员能从学生一页通看到学生基础信息、当前状态和近期待处理记录。
- 所有新增导出能力继续受权限控制，并写入审计日志。
- 不新增家长端、财务、课表或成绩等大模块，避免 V1.5 失焦。

## V2 路线

V2 面向“完整校务平台”的中长期扩展，应在 V1.5 稳定后按学校优先级逐项立项。

| 方向 | 建议能力 | 适合进入 V2 的原因 |
| --- | --- | --- |
| 教务课表 | 课程、任课关系、班级课表、教师课表、调课记录 | 涉及大量结构关系和日常流程，不适合塞进 V1.5 |
| 考试成绩 | 考试批次、成绩导入、成绩分析、成绩单、班级/年级统计 | 对领导决策价值高，但数据敏感且需要独立验收 |
| 德育成长 | 奖惩记录、重点学生跟踪、班级量化趋势、成长档案 | 可复用常规检查基础，但业务口径需先和学校确认 |
| 宿舍生活 | 宿舍楼、寝室、床位、晚寝检查、住宿生名单、宿舍量化 | 当前已有宿舍号搜索需求，可逐步扩展 |
| 招生分班 | 新生登记、材料收集、预分班、正式分班、入学档案转正 | 学期节点性强，适合单独做版本规划 |
| 家校/学生端 | 家长或学生查询、公告、考勤、成绩、消息通知 | 涉及账号发放、隐私、外部访问和消息触达，需要更高安全准备 |

## 暂缓清单

以下能力不是没有价值，而是不适合当前单校内网 V1 后续立即建设。

| 暂缓方向 | 暂缓原因 | 重新评估条件 |
| --- | --- | --- |
| 财务缴费 | 涉及支付、票据、对账和财务制度 | 学校明确要求并能提供财务流程负责人 |
| 资产管理 | 与当前师生档案和检查闭环关联弱 | 现有校务模块稳定，并出现明确资产台账痛点 |
| 食堂管理 | 需要刷卡、消费、供应商或硬件系统对接 | 学校已有食堂系统接口或明确替换计划 |
| 复杂 LMS | 课程资源、作业、在线学习会迅速扩大范围 | 学校明确要建设教学平台而非校务后台 |
| 多校 SaaS | 当前定位是单校内网私有部署 | 出现多校统一部署需求和数据隔离预算 |
| 重型 AI 分析 | 数据量、数据质量和治理边界还未稳定 | V1.5 数据质量、报表和权限审计稳定后再评估 |

## 决策门槛

进入 V1.5 开发前，建议先完成三件事：

1. 确认第一轮真实数据试点范围，是继续高二还是扩到全校部分数据。
2. 确认 V1.5 首批只做 `数据质量 + 导入预检 + 常用导出模板 + 学生一页通`，还是加入轻量考勤。
3. 确认是否需要将敏感字段脱敏、导出原因和 Bootstrap fallback 收紧列为上线前硬要求。
