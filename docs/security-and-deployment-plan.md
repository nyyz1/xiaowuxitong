# 数据安全与部署计划

## 基本原则

- 真实师生数据只放在学校控制的内网或私有服务器环境中。
- 当前试点先保留原有 Excel 流程作为兜底，不做一次性全量切换。
- 权限边界必须在后端执行，不能只依赖页面隐藏按钮。
- 账号、导出、误录删除和敏感变更要留下审计记录。
- 试点阶段优先证明“可控、可回退、可验收”，再逐步扩大使用范围。

## 当前试点部署边界

当前已获批的试点前提是：

- 先在高二年级试点，不直接切全校真实数据。
- Web 应用部署在当前这台学校控制的试点电脑上。
- PostgreSQL 也部署在这台试点电脑的 `D` 盘本地环境中。
- 应用与数据库都放在学校网络边界内，不依赖公网数据库。
- 其他办公电脑通过试点电脑的校内网 IP 访问 Web 应用。
- PostgreSQL 优先只对本机开放，不直接暴露给其他办公电脑。
- 如学校允许 Docker，可以继续使用容器化方式；如学校要求原生安装，则按原生 Node.js + PostgreSQL 方式落地。

## 角色与权限模型

当前试点采用以下角色模型：

- 系统管理员：唯一最高权限账号，负责用户、权限、学校结构和全量业务能力。
- 校领导：查看全校师生、检查记录与检查统计，并执行校级导入导出。
- 年级管理员：必须绑定一个负责年级，只能访问该年级的学生、检查和统计数据。
- 数据管理员：负责全校教师与学生档案的维护、导入和导出。
- 常规检查员：负责检查分类、项目、记录、误录纠错和统计导出。

当前系统已落实的安全边界包括：

- 数据库账号登录与密码哈希存储
- 账号启用、停用与管理员重置密码
- 导出审计日志
- 用户管理相关审计日志
- 年级管理员的服务端年级范围限制

## 试点数据策略

第一阶段建议只使用以下任一数据来源：

- 内置演示数据
- 脱敏样例数据
- 单个年级或单个部门的小范围真实数据

不建议在试点第一天直接导入全校真实数据。

## 备份策略

试点阶段建议：

- PostgreSQL 至少每日自动备份一次
- 在重要导入、权限调整或演示前手动备份一次
- 至少保留 7 天备份
- 在目标服务器上完成至少一次恢复演练

正式扩大范围前，需要明确：

- 备份文件保存位置
- 备份检查负责人
- 恢复执行负责人
- 恢复时长预期

## 上线前检查

进入真实试点前，至少确认：

- PostgreSQL 可连接
- Prisma schema 已同步
- 演示种子可写入
- 试点电脑的校内网访问地址可从其他办公电脑打开
- 数据库用户可正常登录
- 校领导与年级管理员的权限边界符合试点要求
- 师生档案可筛选、导出、下载模板，并支持有权限账号修改或删除误录档案
- 常规检查可录入、修改、删除误录和查询
- 检查统计可导出 Excel/CSV
- `AuditLog` 可写入，并覆盖导出、用户管理和误录删除等敏感动作
- 备份与恢复流程已演练

## 当前限制

- 当前仍未导入学校真实师生数据
- 正式投产前仍需根据学校账号安全要求调整默认演示密码与兜底账号策略
- 机器级、无人登录即可启动的 Windows 服务或计划任务仍需要管理员权限；当前已验证的是当前 Windows 用户登录后自动启动

## 建议下一步

当前试点电脑上的 PostgreSQL 服务、Web 应用、浏览器烟测、备份恢复演练和 2026-05-01 业务页恢复复测已经完成。
下一步应先决定是否进入单年级或单部门的小范围真实数据试点，并在导入真实数据前更换默认演示密码、确认备份责任人和恢复流程。
